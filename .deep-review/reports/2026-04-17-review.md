## 🔴 Critical — Ship Blockers

### X-1. 번역쌍 정규식이 base 파일(`README.md`)을 누락

**3/3 리뷰어 일치.** 이것이 없으면 **self-corruption 버그(C-1) 해결 실패**.

- 스펙 §4.1 정규식: `^(?<basename>[^.]+)\.(?<locale>[a-z]{2}(_[A-Z]{2})?)\.md$`
- 문제: `README.md`는 locale 세그먼트가 없어 이 정규식에 매치 **안 됨**. 그룹은 `{README.ko.md, README.ja.md}` 뿐이고 `README.md`는 그룹에 없음.
- 결과: 스펙이 명시한 예시(`README` 그룹 = {`README.md`, `README.ko.md`, ...})가 실제로 성립 안 함. `README.md ↔ README.ko.md` 쌍의 27줄 JSON 블록은 여전히 "다른 그룹의 중복"으로 분류되어 garden이 삭제 제안.

**권장 수정**:
```
locale 세그먼트를 optional로 처리:
1. 파일명에서 `.{locale}.md`를 벗겨 basename 추출
2. 동일 basename을 가진 모든 `.md` 파일(locale 없는 원본 포함)을 한 그룹
3. 그룹 내부 중복은 audit-only

정규식 대체: 
- 원본 매치: `^(.+)\.md$` (basename 후보)
- locale 변형: basename `foo` + locale segment = `foo.{ko|ja|zh_CN|zh-CN|pt_BR|...}.md`
- 그룹핑 로직: Python/Node 유사 pseudocode로 스펙에 직접 기술
```

추가로 하이픈 locale(`zh-CN`) 지원을 `[a-z]{2}([_-][A-Z]{2})?`로 확장.

## 🟠 High — 병합 전 해결 권장

### X-2. `worktree_hash`가 untracked 파일 미반영

**3/3 리뷰어 일치.**

- 스펙 §5.1: `worktree_hash = git diff HEAD | sha1sum`
- 문제: `git diff HEAD`는 **tracked** 변경만 보여줌. untracked 신규 파일(`?? docs/new-doc.md`)은 해시에 반영 안 됨.
- 영향: 새 doc 추가 후 10분 이내 garden/audit 실행 시 **캐시 재사용** → 새 문서가 스캔 대상에서 누락. H-1의 근본 해결 실패.
- **실제 증거**: 현재 deep-docs 레포가 `?? docs/backlog-2026-04-16.md` 상태. 스펙대로 구현해도 이 상태에서 hash는 "clean"으로 기록됨.

**권장 수정**: `worktree_hash`를 다음으로 변경:
```bash
{ git diff HEAD; git status --porcelain --untracked-files=all; } | sha1sum
```
또는 untracked scan-target 파일의 목록+mtime을 결합한 해시 추가.

### X-3. 코드펜스 dead-ref 필터 누락 / 경계 처리 모호

**3/3 리뷰어 일치 (Opus Warning + Codex P2 + Codex Adv High).**

스펙 §4.3(backtick 필터)은 inline backtick만 다루고, 코드펜스 **내부**의 참조(import문, `src/foo.ts` 예시 경로, CLI 예시)를 제외한다는 규칙이 없음. §4.2는 중복 탐지에서 fenced block을 제거하지만, dead-ref 추출 쪽 Step 2는 그대로 "코드블록 내 import문" 추출을 유지.

**게다가 Codex는 새 부작용 지적**: 코드펜스를 선-제거한 후 남은 prose가 concatenation되어, 원래 **비인접**이었던 3줄이 인접하게 되어 false-positive duplicate 발생 가능.

**권장 수정**:
1. §4.3 조건에 추가: "코드펜스(```) 내부의 참조는 dead-ref 후보에서 제외, audit-only로만 기록"
2. §4.2에 "코드펜스를 제거 시 인접 prose 사이에 **boundary marker**(예: sentinel 라인)를 삽입하여 경계를 넘어선 중복 탐지 방지"
3. 또는 per-segment hashing — 각 fenced block 사이 prose를 별도 세그먼트로 분리 후 각각 해시

### X-4. CLI whitelist 비교 로직 모호

**2/3 리뷰어 일치 (Opus W-2, Codex P2).**

- 스펙 §4.4 whitelist: `git, npm, pnpm, ...` 바이너리 이름
- 문제: 문서에서 추출되는 참조는 `git log -1 --format=%aI`, `npm run build`, `python manage.py` 같은 **전체 명령**. 스펙은 "합집합에 없을 때만 stale"이라 했지만, 전체 문자열과 바이너리 이름을 단순 비교하면 전부 stale로 flag됨.

**권장 수정**: §4.4에 토큰화 규칙 추가:
1. 추출된 CLI 참조에서 첫 토큰을 바이너리로 간주
2. `npm run X`, `make X` 같은 패턴은 2번째 토큰을 script/target으로 분리
3. 매칭: `(binary in whitelist) OR ((binary, script) in package.json/Makefile)`

- **권장**: 다음 중 하나

  3. §2 문구 수정 + CHANGELOG에 Breaking 섹션 추가

### CX-2. Size threshold boundary off-by-one

- 스펙 §6.1: scan 경고 `≥300` (README), audit 만점 `≤300`. **정확히 300줄이면 둘 다 매치** → 경고 + 만점 동시 발생, 논리 모순.
- **권장**: 한쪽을 strict 부등호로:
